我的云上 ECS 龙虾之旅：国内 ECS 玩 OpenClaw 的避坑与技巧

2026-02-26

本文不是保姆级安装教程，看到有人说在ecs上面安装openclaw是地狱级难度，如果是国内的ecs更是地狱级（网络问题），所以我把我这段时间在阿里云 ECS（2G 内存为主）上折腾 OpenClaw 的踩坑整理成一些经验；如果你没有安装的问题，建议直接跳到最后开想法即可；另外其实也建议为了降低安装难度，直接电脑安装（最好独立的）， 或者直接买云服务提供好的openclaw镜像，或者用国外vps；

1 背景：为什么我把 OpenClaw 放到云上

我想玩 OpenClaw 的原因很简单：AI 这波进展太快了，很多东西看起来很热闹，但本质更像“玩具”——可能一周就没消息了。反过来，如果一个东西一个月了还在被大量讨论，而且生态越长越像样，那它大概率就不是昙花一现，是真的值得下场玩一玩。

OpenClaw 给我的感觉就是后者：它的生态已经不太像“玩具”了，很多事情确实能交给它跑。但我一直有两个顾虑：

  • 本机安全：不太想把这么一个高风险的agent放在我本地，而且还不是我自己开发的。还是不信任的问题。
  • 现实条件：我手上刚好有台阿里云 ECS，而且通常是 2G 内存这种“最容易踩坑”的规格——官方教程照着装，直接把2g内存干重启。另外在openclaw刚发布的时候还只是运行卡死，现在直接安装卡死了

所以这篇文章的目标是：

  • 你按这篇能在 2G ECS 上 装上、跑通、稳定起来；
  • 你知道 哪里容易卡死、怎么避坑、怎么排错
  • 分享一下我自己使用下来的一些心得；

2 安装步骤：最短路径（先能装、能跑、能自救）

整个安装的核心是，你先配置一个coding agent，让它去帮你安装龙虾，配置龙虾，拯救龙虾。在低内存 ECS 上，我更推荐你把这些“杂活”交给 Codex / Claude Code（CC）/ OpenCode 去做（它们在服务器上执行命令、改文件、看日志都更顺）。另外这里面codex尤其核心，因为codex是rust，占内存小，量还大，模型还强；

2.0 先把远程运维链路搭好（tmux + Happy + Codex/CC/OpenCode）

为什么要先做这个：

我后面所有“安装/修复/排障”，核心思路都不是自己手敲命令行，而是让 Codex / CC / OpenCode 在服务器上代劳：它们负责跑命令、改配置、看日志、重启服务；我只盯关键点和结果。

但云端这套东西还有一个现实问题：连接会断，OpenClaw（龙虾）本身也可能会挂。所以我会再加一层“不断线/可救火”的运维底座：

  • tmux：把远程操作放进 tmux 里，手机切后台/网络抖动/SSH 掉线时，不至于把正在进行的操作和上下文一起断掉。
  • Happy：当龙虾挂了、gateway 需要重启、配置需要紧急改一下时，我可以用手机 App 通过 Happy 去“控制 Codex/CC/OpenCode”，在云机上把龙虾拉起来（比如重启服务、改代理、关掉出问题的组件）。另外我还设置了在机器重启的时候自动创建tmux启动codex，确保我有一个入口继续操作

工具与模型怎么选（我自己的经验）：

  • 工具层面：Codex 之外，你也可以用 Claude Code（CC）、OpenCode 这类 CLI 方案配合使用（无桌面/手机端都很方便）。
  • 模型层面：你当然可以接入一些免费/便宜的模型（比如 Kimi、GLM-5、MiniMax）来做“轻量活”；但在我这里，关键链路最好用 SOTA（比如 Codex 或 Claude Code 这类），因为低内存 + 弱网络 + 真实系统环境，容错能力很重要，省下来的时间往往比省下来的 token 更值钱。
  • Codex 这边我更推荐 Device Mode 做授权（无桌面最省事）。
  • Happy 这边我主要用来做“手机端不断线的运维入口”，和 tmux 配合效果最好。

2.1 先打通网络：代理 + HTTP(S)_PROXY

为什么：

在国内的ECS上面玩，最大的问题就是网络问题了，你后面会遇到 Git clone、npm、Telegram 媒体下载、甚至一些站点访问。网络没打通，所有问题会叠在一起，排到最后很痛苦。

我一般这么弄：

  • 在服务器上装好 Clash（让前面那一步的coding agent帮你装），并确保本地监听端口可用（例如 127.0.0.1:7890）。
  • 关键不是“你设置了代理”，而是：npm / git / Telegram 媒体下载，都稳定走同一条出站路径。
  • 如果你用 Codex / CC / OpenCode：直接让它在服务器上装代理、写环境变量、并做连通性验证即可。

怎么判断外网通了（任选其一）：

  • curl -I https://registry.npmjs.org/ 能返回 HTTP 响应头；
  • 或者 curl -I https://www.google.com/ 能返回 HTTP 响应头（用来快速判断外网是否打通）；
  • 或者 npm view openclaw version 能正常返回。

如果还是不通：

  • curl 能通但 Telegram 图片不通：很可能是“文字 API 可用，但媒体下载链路走了不稳定的路径”，后面 Telegram 章节我会把“显式 proxy 写进 openclaw.json”的做法放进去。
  • 代理“有时生效有时不生效”：优先检查进程实际拿到的环境变量是否一致；必要时把 proxy 写进对应服务/应用自己的配置里，而不是只靠 shell 里的 export。

2.2 安装 OpenClaw：2G 避坑版（固定版本 + 去 optional）

我遇到的典型坑：

按官方方式（无论 npm 还是 curl）在 2G 机器上装，会出现 RAM 99% → 机器卡死 → 只能重启。

先说明一下：官方/社区其实已经有一堆安装指令，正常情况下你完全可以先照着跑（甚至很多机器一次就能装好）。 但我自己当时的经验是：至少在 2026-02-19 这个时间点，我在 2G 内存 ECS 上反复尝试安装，基本都会卡到 OOM（装到一半机器直接不响应，只能重启）。

这里补一个“卡死到底卡在哪”的细节：我当时让 Codex 跟着安装输出和 npm 日志去定位，最后发现机器最容易死在某些依赖的 postinstall 阶段——尤其是安装本地推理相关的组件时（我当时看到最明显的是 node-llama-cpp / llama.cpp 这一类）。这种依赖要么会下载较大的文件，要么会跑原生编译/构建脚本，对 2G 内存 + 弱网络 特别不友好：一旦触发重试或并发，内存就会被顶满。

所以我最后才收敛出一条“2G 生存版”的兜底路径：清残留 + 固定到已验证版本 + 安装时去掉 optional 依赖 + 放宽 peer dependency 检查。

先解释一下我为什么这么装（这部分我认为比“抄命令”更重要）：

  • 为什么我会选择 固定版本（作为兜底）：在低内存机器上，最怕“你装的不是 OpenClaw，而是安装过程里的随机失败”。当官方/社区安装在 2G 上反复卡死时，固定到我验证过的版本，能把变量从 N 个降到 1 个（先跑通再说）。如果你按官方装得动，就没必要一开始就固定版本。
  • 为什么要 --omit optional：npm 的 optionalDependencies 往往是“不是每个人都需要，但一旦装就可能触发重依赖/原生编译”的东西（典型就是某些本地推理/加速相关依赖会跑 postinstall）。在 2G ECS 上，这类依赖更容易把你拖进 OOM 或网络抖动导致的失败。如果你当前只需要 Telegram + 基础能力，先跳过 optional 往往更稳。
    • 代价：某些可选能力可能就不会装上（后面真要用，再单独补装/换更大规格验证）。
  • 为什么要 --legacy-peer-deps：从 npm v7 开始，peer dependency 更严格，遇到冲突会直接失败；而“失败一次再重试”在 2G 机器上，成本很高。--legacy-peer-deps 相当于让 npm 用更宽松的策略（类似 npm v6 的行为），优先把东西装起来。
    • 代价：你可能会得到一个“不是最完美的依赖树”；但对“先跑通网关/先把链路打通”来说，它是非常实用的交换。
    • 再多解释一句：peer dependency 通常表示“这个包希望你在上层已经有某个依赖，并且版本要匹配”。npm v7+ 会尝试把 peer 也一起解出来，冲突时就中止；而在低内存 ECS 上，这种“解依赖失败 → 重跑 → 又失败”的体验特别差，所以我宁愿先把路打通。

一句话总结：这几个参数不是“正确姿势”，而是低内存环境的生存策略——先把 OpenClaw 跑起来，后面要更“干净/更严格/更全功能”，再开一条线慢慢升级。

我当时是这么做的：

建议顺序是：

  1. 先按官方/社区的安装方式试一次（很多机器是能一次成功的）。
  2. 如果你在 2G 上装到 OOM 卡死/反复失败，再用下面这条“2G 生存版”兜底。

下面这些操作我建议都让 Codex / CC / OpenCode 在 tmux 里执行（你只盯着“有没有装成/有没有跑通”就行）。如果你手动做，也按这个顺序来。

  1. （可选）先看一下自己现在装的版本/是否残留
npm list -g openclaw --depth 0
  1. 如果你之前装坏过/更新中断过，先清理残留目录（避免 ENOTEMPTY）

注意：下面命令会删除全局安装目录里的 openclaw，仅在你确认“它已经损坏/装坏了”的前提下执行。

sudo rm -rf /usr/lib/node_modules/openclaw
  1. 固定到我验证过的版本安装（2026.2.19-2）
npm install --global openclaw@2026.2.19-2 --omit optional --legacy-peer-deps

怎么确认装好了：

  • npm list -g openclaw --depth 0 能看到 openclaw@2026.2.19-2；
  • 后续你能正常启动网关（见 Telegram 小节里的 openclaw gateway restart）。

如果翻车了：

  • 报 ENOTEMPTY（目录非空导致卸载/覆盖失败）：按上面的残留清理再来一次。
  • 你装 latest 失败：我这边遇到过 openclaw@2026.2.21-2 在 Node 22 环境触发 @discordjs/opus 的预编译缺失，回退编译又受网络波动影响导致中断。
    • 处理：先用我上面的“固定版本命令”跑通；如果你必须升级，再单独开一节做升级验证。

2.3 Telegram：先跑通，再谈多会话管理

我建议你先把目标定到“私聊能正常收发”，再去做群聊/多会话隔离。因为群聊往往会引入 allowlist、权限策略等变量，让你误判成“机器人坏了”。

2.3.1 先把 Telegram 的网络链路搞稳（尤其是发图）

我遇到过一个很典型的问题：文字能发，但发图后助手看不到图片，日志会出现 MediaFetchError。最后的关键修复是：把 Telegram 通道的 proxy 显式写死，别只靠系统环境变量。

**我当时这么修：**在 /root/.openclaw/openclaw.json 里加入（或合并）：

{
  "channels": {
    "telegram": {
      "proxy": "http://127.0.0.1:7890",
      "network": {
        "autoSelectFamily": false
      }
    }
  }
}

然后重启网关（让 Codex / CC / OpenCode 跑也行）：openclaw gateway restart

怎么验证：

  • 重新发一张图，助手能正常“看图”并回答；
  • 或者你至少看不到 MediaFetchError 反复出现。

如果还不行：

  • 仍然偶发失败：优先检查本机 Clash 端口是否真的可用、以及是否存在 IPv6 路径抖动（上面 autoSelectFamily: false 就是我用来压这个抖动的）。

2.3.2 多会话/多主题管理（避免“所有聊天都糊成一团”）

我比较推荐两种方式（二选一）：

  • Telegram 频道/群 + 邀机器人 + 用主题/@ 做隔离：不同主题对应不同上下文，减少串台。
  • 用 Discord 做隔离：把“工作流讨论”和“日常闲聊”拆开（更干净）。

2.4 浏览器能力：browserless + Web Fetch（2G 友好）

对 Agent 来说，“能抓网页内容”和“能像人一样打开网页”其实是很关键的两类能力：前者适合做资料检索/核查，后者适合做动态页面、需要点击/滚动/截图的任务。很多时候你以为自己只需要一个聊天机器人，但真正跑起来会发现：Web Fetch + 浏览器能力经常才是把任务跑通的关键。

但在 2G ECS 上本地装 Chrome/Chromium + Playwright，我个人不推荐：太吃内存，稳定性也更差。

我的做法是把“重浏览器”从 ECS 上挪走：用 browserless.io 这种托管浏览器当替代（必要时再配合 Web Fetch 做轻量抓取）。我写本文时看到它有免费额度（约 1000 次/月，具体以官网为准），足够你先把链路跑通。

我一般这么接：

  • 去 browserless 申请一个 API Token。
  • 让 Codex / CC / OpenCode 在服务器上把这个 Token 配到 OpenClaw 的 browserless 配置里（让龙虾自己把配置写好、重启好，你不用手敲）。
  • 任务层面优先级：能用 Web Fetch 就先用 Web Fetch；遇到动态页面/必须截图再走 browserless。

怎么验收：

  • 在 Telegram 里让龙虾打开 https://www.baidu.com/ 并回一张截图给你：能成功截图就算链路打通。

如果翻车：

  • 报 timed out after 1500ms：我遇到过“browserless 实际可用，但被 1500ms 的硬编码超时误判”的情况。
    • 处理：我单独记了一份排查记录（包含临时热修思路与回归检查）：/Users/breeze/personal/ai-video/knowledge-base/30_Resources/OpenClaw/browserless-timeout-1500ms-fix-2026-02-25.md
  • 能访问但截图空白/加载不全：优先排查代理链路是否稳定（和前面“网络”“Telegram”那两节的思路类似），以及把超时参数调大再试。

3 真实场景：我现在用它做什么

3.1 个人知识库（PARA）+ Git 自动推送/多端同步

以前我收藏内容很散：Apple Notes、微信收藏、各种 App 里。现在我在 ECS 上做了一个 PARA 结构的文件系统，把看到的内容丢进去，再让它帮我做一些整理。配上 Git 之后，多端同步也更顺。 现在不管是日常代办，看到的AI咨询，路上想到的电子，都是直接往里面一丢；它就可以按照（PARA：Projects/Areas/Resources/Archive）这四层目录结构帮我分门归类好；而且能自动commit和push；甚至还有自动提醒归类；这一整套的实现都是龙虾自己实现的，我让它装了一个gh的cli（github的），它自己就玩转了；

3.2 定时访问与信息核查（check news / polymarket 等）

我更喜欢把它当“流程型任务”的执行器：比如定时访问某些站点、核查一条新闻是否属实（比如最近看到有自媒体说美国说要在2.23号和24号攻击伊朗，这时候让龙虾自己去polymarket 看一眼赔率变化）。你只需要关心最后输出的结果。通过这种方式就能把你日常看框

3.3 围绕个人事项去做

线下聚会里听到有人用类似的 agent 去做 App Store release 的多语言翻译、以及批量回复用户评论。每个人的“高频痛点”不一样，但思路是一致的：把自己手上的事情尽可能AI和自动化，让这些

4 一些想法

4.1 OpenClaw 更像新时代的“电子宠物”

OpenClaw 可能更像一个新时代的电子宠物，或者说它把“私人助理”这层帷幕真正拉开了一点点。

我小时候看《宠物小精灵》的时候，会想象“要是我也能有一只皮卡丘/喷火龙就好了”。你想想那个设定其实特别像今天的 agent：它有一个大脑（你给它什么模型），有一套技能（你能给它装各种能力/插件），然后它会在你的真实生活里帮你跑腿、打怪、升级。

你给它装技能、让它学会新东西，这件事本身就很像“打怪升级”。甚至你还可以让它去参加比赛——比如我最近就看到社区有人搞过 OpenClaw 相关的作文/创作类活动（链接：OpenClaw 作文任务），然后我也让我的龙虾去参加了，拿了一个86分，不过第一名已经卷到98分去了，不过本身就是个尝试；

然后有人可能会说，好像装了openclaw也不知道让它干嘛，但是我自己的感受下来，就是上面给它装各种工具，用它去维护我的知识库，看着它提醒我各种事项，折腾的感觉也很好，总之感觉还是得先有，有了之后可以尝试适合它的任务就让它去干；本身也是探索和进化的过程；另外大部分的安装我都是让它自己干，只有部分比较难的，我才启动codex去调试安装它；

如果在玩的过程中，发现这个2g的ecs内存开始限制它的能力了，那就真的有必要给它升级一套身体了！希望下次还能更新到这里

4.2 大脑要顶配，成本要分层（顺便说个我踩过的坑）

一个好的私人助理，大脑真的最重要。很多人觉得“OpenClaw 也就那样，干啥都傻傻的”，我现在的体感是——很可能不是 OpenClaw 的问题，而是你给它配了一个太弱/免费的模型。

我的建议是分两层：

  • 关键链路（需要稳、需要交付质量的）尽量用最好的模型顶上去，比如 ChatGPT / Claude Code 这一档。
  • 执行层（能切碎、能重试、能容错的）再去配一些性价比更高的模型，慢慢把成本压下来。

先把“交付质量”跑出来，再去谈“怎么省钱”，心态会舒服很多。

最后补一个我自己踩过的成本坑（也是我后来很强调“分层”的原因）：我曾经因为担心内存过大导致死机，让它写了一个 10 分钟一次的检测脚本。然后两天多的时间直接把我的codex额度全跑过了，总过调用了大几百次，然后用老了接近一个亿的token，主要还是定时任务太频繁、还会失败重试，而且一直往同一个会话里滚，导致上下文越来越大，后面每次调用都会更贵、更慢。

粗略估算下来，这个 10 分钟脚本本身就跑了几千万 token（再叠加日常对话/其他任务），最后一起把总量顶爆。后来我做的调整是：

  • 把频率降下来：改成 2–3 小时一次；
  • 不要把所有定时任务都绑定到同一个会话无限增长；
  • 失败要有熔断/上限，别无限重试。

4.3 产品体验也很重要

一开始我也觉得 openclaw 这一套，不就是以前 codex 或者 claude code 直接控制整个计算机？但是当你发现你能用 telegram、discord、钉钉、飞书去连接的时候，就真的不一样了。另外再看看它的 soul.md，原来它的设计里面还有灵魂：当你不再是冷冰冰地让它写代码，而是看着它干各种事情，最后还会跟你说一句晚安的时候——这不就是我们小时候想要的那只皮卡丘吗；
